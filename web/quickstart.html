Quick start

Ship a governed agent in 60 seconds

Three commands and a YAML file. The proxy runs locally, wraps any MCP server, and enforces policy before the upstream is ever contacted.

Prerequisites

  • Go 1.25+ with eunox-mcp available on your PATH
  • An MCP host: Claude Desktop, Cursor, Windsurf, or any agent that speaks MCP
  • An upstream MCP server you want to govern (e.g. @modelcontextprotocol/server-filesystem — requires Node.js & npm)
Note. Install or build the eunox-mcp binary once, then reuse it from Claude Desktop, Cursor, Windsurf, scripts, and CI.

1. Install

Install the OSS CLI as a Go binary:

go install github.com/edgeobs/eunox/cmd/eunox-mcp@latest
eunox-mcp --version

You can also download a release build if you prefer not to install from source.

2. Author a policy

A policy is a single YAML file describing which tools are allowed and the conditions on each call. Save the following as eunox.policy.yaml:

agentId: filesystem-agent
name:    Filesystem Agent
version: 0.1.0

requiredCapabilities:
  - resource: read_file
    actions: [call]
    conditions:
      - type: allowedExtensions
        extensions: [".csv", ".json", ".txt", ".md"]
      - type: maxCalls
        count: 50
        windowSeconds: 60

  - resource: list_directory
    actions: [call]

The Go reimplementation keeps the same AgentCapabilityManifest shape across the local proxy, the hosted services, and the in-process runtime paths.

3. Validate the policy

eunox-mcp validate ./eunox.policy.yaml

Validation runs without contacting any server and rejects unknown condition types and malformed YAML up front.

4. Run the proxy

Wrap any MCP server with the proxy. Everything after -- is the upstream command:

eunox-mcp proxy \
  --policy ./eunox.policy.yaml \
  -- npx -y @modelcontextprotocol/server-filesystem /data

The proxy speaks MCP on stdio in both directions and is transparent to clients.

5. Try it from a host

Point your MCP host (or any client) at the proxy command above. From Claude Desktop, attempt to read /data/keys.pem — the proxy denies it before the upstream is contacted, and the agent receives a structured CapabilityDenied error.

6. Inspect the audit log

Every decision — allow or deny — is recorded as a signed OCSF API Activity event:

tail -F ~/.eunox/audit.jsonl | jq

For aggregated denial counts, run:

eunox-mcp stats

This prints an ASCII histogram grouped by condition type and denial code, including rotated log files.

Claude Desktop / Cursor / Windsurf

Add the proxy as an mcpServers entry in claude_desktop_config.json (or the equivalent file for Cursor / Windsurf):

{
  "mcpServers": {
    "filesystem-governed": {
      "command": "eunox-mcp",
      "args": [
        "proxy",
        "--policy", "/absolute/path/to/eunox.policy.yaml",
        "--",
        "npx", "-y", "@modelcontextprotocol/server-filesystem", "/data"
      ]
    }
  }
}

Without --policy the proxy is transparent — handy for capturing an audit trail before you write any rules.

HTTP transport · Go agents

For agents that connect over HTTP rather than stdio, run the proxy as an HTTP server:

eunox-mcp proxy \
  --transport http --port 7391 \
  --policy ./eunox.policy.yaml \
  -- ./my-mcp-server

Connect your client to http://127.0.0.1:7391/mcp. The same policy file works on both transports — and ipRange conditions become enforceable here.

In-process Go runtime

If your agent already runs in Go, use the Agent Runtime and adapter layer instead of a separate JavaScript package:

runtime, _ := agentruntime.New(cfg)
adapter := adapters.NewLangChainAdapter(runtime, "session-1",
    adapters.Tool{Name: "query_db", Description: "Run a read-only SQL query"},
)

result := adapter.CallWithArgs(context.Background(), "query_db", map[string]interface{}{
    "sql": "SELECT * FROM users",
})

See the Go agent runtime docs for the full runtime configuration and adapter examples.
