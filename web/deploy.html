OSS — Apache-2.0 (no server required)

The @eunox/mcp package runs entirely in-process. All enforcement, call counting, and audit logging happens locally — no Redis, no Postgres, no gateway process.

Install

npm install -g @eunox/mcp

Write a policy

# policy.yaml
version: "1"

capabilities:
  - id: allow-read-only
    tools:
      - name: read_file
      - name: list_directory
    conditions:
      - type: path_prefix
        prefix: "/home/user/projects"
    audit:
      required: true

  - id: deny-writes
    tools:
      - name: write_file
      - name: delete_file
    effect: deny

The policy engine is default-deny: any tool not covered by an allow capability is blocked. See capability-manifest-guide.md for all condition types.

Wire into Claude Desktop

// ~/Library/Application Support/Claude/claude_desktop_config.json
{
  "mcpServers": {
    "filesystem-governed": {
      "command": "npx",
      "args": [
        "@eunox/mcp",
        "--policy", "/path/to/policy.yaml",
        "--upstream", "npx",
        "--upstream-args",
        "@modelcontextprotocol/server-filesystem,/home/user/projects"
      ]
    }
  }
}

Enable the local audit log

EUNOX_AUDIT_LOG_PATH=./audit.jsonl eunox-mcp serve --policy ./policy.yaml ...

Records are HMAC-chained for tamper-evidence. Verify the chain with eunox-mcp validate-log ./audit.jsonl.

Upgrade when you need shared state

eunox-mcp upgrade-to-hosted \
  --gateway-url https://your-gateway.example.com \
  --api-key <your-issued-jwt>

See the full OSS deployment guide and upgrade-to-hosted.md for the complete walkthrough.

Self-Host — BSL 1.1 (BYO infrastructure)

The self-host tier deploys the full gateway stack under your control: capability issuer, tool gateway, minter, posture emitter, and enterprise services. You manage Redis, Postgres, and a cloud KMS.

License. BSL 1.1 — non-competing use; converts to Apache-2.0 four years after each release. Review the LICENSE before deploying in a competing product.

What you need

  • Docker ≥ 24 and Docker Compose ≥ 2 (or Kubernetes with Helm)
  • A cloud KMS account: AWS KMS, Azure Key Vault, or GCP Cloud KMS
  • Redis ≥ 6.2 (Sentinel or Cluster for production HA)
  • Postgres ≥ 14

Development setup (Docker Compose, no Redis/Postgres)

# /srv/eunox/docker-compose.dev.yml
version: "3.9"
services:
  capability-issuer:
    image: ghcr.io/edgeobs/eunox/issuer:1.0.0
    env_file: /srv/eunox/issuer.env
    volumes: ["/srv/eunox/policies:/app/policies:ro"]
    ports: ["3001:3001"]

  tool-gateway:
    image: ghcr.io/edgeobs/eunox/gateway:1.0.0
    env_file: /srv/eunox/gateway.env
    volumes: ["/srv/eunox/keys:/app/keys:ro"]
    depends_on: [capability-issuer]
    ports: ["3002:3002", "127.0.0.1:3003:3003"]
docker compose -f /srv/eunox/docker-compose.dev.yml up -d

Production deployment (Helm)

helm install eunox k8s/helm/eunox/ \
  --namespace eunox-system \
  --create-namespace \
  -f k8s/helm/eunox/values-prod.yaml

The Helm umbrella chart deploys all six services with individual HorizontalPodAutoscalers. For EKS and GKE cloud-specific guides, see deploy-eks.md and deploy-gke.md.

Redis HA requirement

In production mode (NODE_ENV=production), the gateway performs a fatal startup check: if any configured Redis URL points to a single-node instance, the service refuses to start. Configure Redis Sentinel or Redis Cluster for all Redis URLs.

Air-gap deployments

# On a machine with internet access:
./scripts/pull-air-gap-images.sh
# Transfer air-gap-images.tar to the restricted network and load:
docker load < air-gap-images.tar

The full image list is in k8s/air-gap-images.txt.

Wire @eunox/mcp to the gateway

{
  "mcpServers": {
    "my-server-governed": {
      "command": "npx",
      "args": [
        "-y", "@eunox/mcp", "proxy",
        "--enforcer-url", "https://gateway.example.com",
        "--enforcer-api-key", "<your-issued-jwt>",
        "--", "npx", "-y", "@modelcontextprotocol/server-filesystem", "/data"
      ]
    }
  }
}
